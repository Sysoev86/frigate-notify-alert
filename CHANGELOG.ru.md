## [Unreleased]
### Добавлено
- **Режим сигнализации — мгновенный текст после простоя** (`IDLE_ALERT_AFTER`, по умолчанию
  выкл.). Позиционирует бота как лёгкую охранную сигнализацию: если событий не было заданное
  время (например, час), первое событие, нарушившее тишину, шлёт немедленный текст со звуком
  («🚨 Движение после тишины (2 ч) — person @ камера; фото и видео сейчас придут») — *до*
  фото+видео, которым нужно несколько секунд на подготовку. Так возможный несанкционированный
  проход отмечается сразу, не дожидаясь медиа. Срабатывает один раз на событие в обоих путях
  (MQTT и опрос); уважает паузу; события во время паузы всё равно сбрасывают счётчик простоя.
  `0` выключает (по умолчанию) — на существующих установках ничего не меняется.
- **README теперь начинается с сценария «сигнализация»**: события уходят push'ом по MQTT
  (опрос — лишь подстраховка), поэтому попадают в Telegram максимально быстро; `silent: False`
  даёт звуковое оповещение.

### Исправлено
- **Уведомление больше не теряется, когда недоступен только клип.** Если Frigate отметил
  событие готовым, но по факту не отдавал `clip.mp4` в отведённое окно ретраев — терялось
  всё уведомление целиком. Теперь в этом случае в чат уходит компактное **текстовое**
  уведомление («клип пока недоступен — смотри во Frigate», с объектом и камерой), а не
  пустота. Одиночное фото намеренно *не* шлём: Telegram показывает одиночный медиа-элемент
  слишком крупно, поэтому медиа уходит только альбомом (фото+видео) либо не уходит вовсе.
- **Разовый таймаут Telegram больше не роняет уведомление.** Один `Timed out` / сетевой
  сбой при отправке в `api.telegram.org` раньше терял сообщение. Теперь отправка
  повторяется несколько раз с растущей паузой (`TELEGRAM_SEND_RETRIES`,
  `TELEGRAM_SEND_BACKOFF`); устойчивые не-транзиентные ошибки (не тот чат, кривой запрос)
  по-прежнему падают сразу.
### Изменено
- Ошибка «media never became available» теперь пишет, **какой** файл не дошёл (фото/клип)
  и **почему** (HTTP-код / таймаут / недописанный файл) — трекер говорит, что реально
  случилось, а не просто «после N попыток».

## [1.3.4] — 2026-07-16
### Исправлено
- **OOM на огромных клипах.** Длинные события (человек час работает на складе) дают
  клипы на сотни МБ; их скачивание в память убивало процесс OOM-киллером — а боты
  Telegram всё равно не могут загрузить файл больше 50 МБ. Теперь размер клипа ограничен
  (`MAX_CLIP_MB`, по умолчанию 45; проверка по Content-Length и по ходу стриминга).
  События с большим клипом приходят обычным компактным альбомом фото+видео с **обрезанным
  клипом** (первые N секунд, длина считается из битрейта, через API записей Frigate по
  диапазону времени) и подписью; одиночное фото — только самый крайний случай.
- **Конфликт webhook.** Webhook, оставшийся на токене бота (от другого бот-софта), молча
  ломал кнопки паузы бесконечными `409 Conflict`. Контроллер теперь находит и снимает его
  при старте; `doctor` тоже об этом сообщает.
- **Токен бота больше не течёт в логи.** httpx логировал каждый URL запроса вместе с
  токеном — его INFO-логирование приглушено в обоих процессах.

### Добавлено
- **Подсказки по конфигу.** `manage.sh update` приносит новый код, но `config.py` не
  трогает — новые опции оставались невидимыми (один пользователь неделями сидел на
  английском интерфейсе, не зная про `LANG`). Теперь стартовый лог монитора и `doctor`
  перечисляют опции, которых нет в конфиге, и отдельно ловят русскую установку
  (кириллица в именах групп) на английском интерфейсе. Будущие опции подхватятся сами.
- Опциональная отправка ошибок в GlitchTip/Sentry: переменная окружения `GLITCHTIP_DSN`
  (+ `pip install sentry-sdk`); без неё ничего не меняется.

### Внутреннее
- Валидация конфига была продублирована в мониторе и контроллере (и третьим вариантом в
  `doctor`) — вынесена в общий `config_check.py`.

## [1.3.3] — 2026-07-06
### Исправлено
- **Зависший закреплённый статус паузы.** «🔕 пауза до X» обновлялся только по нажатию
  кнопки — истёкшая пауза оставляла пин навсегда. Теперь вотчер снимает его сам в
  течение ~минуты. (Сами уведомления всегда возобновлялись вовремя — монитор проверяет
  истечение самостоятельно.)

### Изменено
- **Панель, зависящая от состояния.** Больше нет вечной кнопки «▶️ Включить» при
  включённых уведомлениях: клавиатура показывает только кнопки паузы, а на паузе —
  «Включить» + продление. Одна панель вместо пары «клавиатура + статус»; закрепляется
  только на время паузы. Старые сообщения подчищаются при обновлении автоматически.

## [1.3.2] — 2026-07-05
### Добавлено
- **`manage.sh doctor`** — самодиагностика всей цепочки одной командой: заглушки в
  конфиге, доступность Frigate API, имена камер/зон (с подсказками при опечатках),
  snapshots/record по камерам, медиа у свежих событий, MQTT-авторизация, токен
  Telegram и доступ к чатам, статус сервисов.
- **Стартовое сообщение**: каждый монитор при запуске шлёт в свой чат одно тихое
  «мониторинг запущен» (камеры, объекты, зоны, версия) — встроенная проверка связки
  бот↔чат при каждом старте. Отключается `"startup_message": False` на группу.
- **Валидация конфига при старте**: понятные ошибки «что именно не так» (заглушки,
  пропущенные поля) вместо трейсбеков — и в мониторе, и в пульте паузы.

## [1.3.1] — 2026-07-05
### Добавлено
- Опция `silent` на группу: `True` (по умолчанию) — беззвучная доставка, как раньше;
  `False` — уведомления приходят со звуком/вибрацией.

### Изменено (внутреннее)
- systemd-юниты генерируются командой `manage.sh install` (отдельные `.service`-файлы
  удалены); юниты вызывают venv-python напрямую.
- Дружелюбные проверки: нет `python3`, подсказка про `python3-venv`, запасной вариант
  без systemd (`manage.sh run`), проверка root для systemd-команд.

## [1.3.0] — 2026-07-05
### Изменено
- **Один скрипт вместо четырёх.** В `manage.sh` добавлены `setup` (первая установка:
  зависимости + юниты + запуск) и `run [группа]` (ручной запуск без systemd).
  `install_deps.sh`, `run_monitor.sh` и `server_setup.sh` удалены — свежая установка

### Исправлено
- Ручной запуск не передавал монитору имя группы (он сразу выходил с подсказкой);
  `manage.sh run` подставляет группу сам, если она одна.

### Удалено
- Разовая команда `migrate` (переезд со старых юнитов, больше не нужна).

## [1.2.0] — 2026-07-05
Релиз-чистка кода. Поведение по умолчанию не изменилось.

### Исправлено
- **Опции конфига теперь реально работают.** `OBJECTS`, `LOG_LEVEL`, `LOG_FORMAT`,
  `STATS_INTERVAL` и `MEDIA_RETRY_ATTEMPTS` были задокументированы, но молча
  игнорировались (значения были зашиты в код). Теперь применяются; отсутствующие
  опции получают прежние значения по умолчанию — старые конфиги работают как раньше.
- **Совместимость с paho-mqtt 2.x.** Свежие установки ставили paho-mqtt 2.x, где
  `mqtt.Client()` без версии callback API падает. Добавлена прослойка — работают
  и 1.x, и 2.x.
- **Медиа скачивается один раз, а не дважды.** Фото и клип скачивались с Frigate
  «для проверки доступности», а потом скачивались заново для отправки.

### Изменено
- **Логи стали сильно тише.** Рутинные дампы опроса (каждое событие каждые 3 секунды)
  перенесены с INFO на DEBUG — лог-файлы больше не растут на мегабайты в час.
  `LOG_LEVEL = "DEBUG"` возвращает подробный вывод.
- Сообщения логов и комментарии в коде — на английском (публичный репозиторий).
- Одна общая HTTP-сессия на монитор вместо новой на каждый запрос.
- Неиспользуемый `MEDIA_WAIT_TIME` заменён на `MEDIA_RETRY_DELAY` (пауза между
  попытками скачивания, по умолчанию 3 с — фактическое поведение то же).

### Добавлено
- Опция `objects` на группу: переопределить глобальный `OBJECTS` для одной группы
  (например, `["person"]` для камеры в помещении).
- README: пример Frigate теперь явно рекомендует
  `record.*.retain.mode: active_objects` (с `motion` маска движения может молча
  убивать клипы — ловушка «фото есть, видео нет»).

## [1.0.0] — 2026-07-05
Первый публичный релиз.

### Возможности
- Мониторинг событий Frigate через MQTT (`frigate/events`), отправка фото + видео
  в Telegram при обнаружении человека/машины.
- Несколько групп камер — каждая в свой чат (`GROUPS`).
- Масштабирование на любое число групп через шаблонный systemd-юнит
  `frigate-telegram@<группа>`; `manage.sh` читает список групп из конфига.
- Фильтр по зонам (`zones`) — слать только когда объект в нужной зоне.
- Пауза уведомлений кнопками в чате (`mute_controller`): клавиатура внизу чата,
  пауза на группу, переживает перезапуск. Включается флагом `mute_controls`.
- Опциональный прокси для Telegram.
- Управление через `manage.sh`.
